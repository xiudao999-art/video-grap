# douyin-download

Drives the locally installed `jiji262/douyin-downloader` (v2.0.0) at
`D:\video-grap\douyin-downloader`. It is **not** on PATH — always invoke via the
project venv python + `run.py`.

## Engine location & env

- Root: `D:/video-grap/douyin-downloader`
- Python: `D:/video-grap/douyin-downloader/.venv/Scripts/python.exe`
- Entry: `D:/video-grap/douyin-downloader/run.py`
- Config: `D:/video-grap/douyin-downloader/config.yml` (already created from example)
- Cookies: `D:/video-grap/douyin-downloader/config/cookies.json` (generated by login)
- Downloads land in `D:/video-grap/douyin-downloader/Downloaded/`

## ⚠️ Environment quirks (this machine)

1. **Bash sandbox blocks network.** Any invocation that touches the network
   (downloads, cookie fetch, `--hot-board`, `--search`) needs
   `dangerouslyDisableSandbox: true` on the Bash tool call.
2. **`dangerouslyDisableSandbox: true` swallows stdout/stderr.** The command
   runs but you see "no output". So always redirect to a log file in a
   Windows-visible path and Read it back:
   ```bash
   cd "D:/video-grap/douyin-downloader" && {
     .venv/Scripts/python.exe run.py --version
   } > D:/video-grap/.run.log 2>&1
   ```
   then `Read D:\video-grap\.run.log`.
3. `/tmp` is msys-only — Windows `python`/Read can't see it. Only use `D:/...`
   paths for any file you need to read back.
4. Chinese console output from `run.py --help` shows as mojibake under the
   bash capture — that's a display issue, the CLI works fine.

## First-run: cookie login (one-time, needs the human)

Downloads require an authenticated Douyin session. The first time, cookies are
placeholders and the run will fail with a login-required error. Do ONE of:

**A. Interactive (recommended):** run a download; on `LoginRequiredError` the
tool auto-opens a Playwright Chromium window. The user logs into douyin.com in
that window, comes back to the terminal and presses Enter. Cookies are written
to `config/cookies.json` and the run retries.

**B. Manual fetch:**
```bash
cd "D:/video-grap/douyin-downloader" && {
  .venv/Scripts/python.exe tools/cookie_fetcher.py
} > D:/video-grap/.login.log 2>&1
```
Same browser-login flow, writes `config/cookies.json`.

Cookies expire. If a later run says 登录态失效 / status 401-ish, re-run the
login flow. `config/cookies.json` is gitignored — do not commit it.

## Common invocations

All examples assume `cd "D:/video-grap/douyin-downloader"` and the venv python.
Wrap in `{ ... } > D:/video-grap/.run.log 2>&1` and remember
`dangerouslyDisableSandbox: true` on the Bash call.

| Goal | Command |
|---|---|
| One video / note / 短链 | `python run.py -u "https://v.douyin.com/xxxx/"` |
| Batch a creator's posts | edit `config.yml` `link:` to the user URL, keep `mode: [post]`, set `number.post`, then `python run.py` |
| Creator's likes | `mode: [like]`, set `number.like` |
| Favorites collections (must be logged in) | link = `/user/self?showTab=favorite_collection`, `mode: [collect]` or `[collectmix]` |
| Live recording | `-u "https://live.douyin.com/{room_id}"` |
| Hot board → JSONL | `python run.py --hot-board 50` |
| Keyword search → JSONL | `python run.py --search "关键词" --search-max 50` |
| REST API server mode | `python run.py --serve --serve-port 8000` (needs `uv pip install fastapi uvicorn`) |
| Show version | `python run.py --version` |
| Verbose logs | add `-v` or `--show-warnings` |

CLI flags override `config.yml`. `-u` can be repeated. Cross-mode dedup by
`aweme_id` is automatic (SQLite + local file dual check).

## Config knobs worth knowing (config.yml)

- `thread` (default 5), `retry_times` (default 3) — concurrency + exponential
  backoff. Keep modest to stay low-risk.
- `proxy: ""` — set to a residential proxy if downloads get region-blocked.
- `browser_fallback.enabled: true` / `headless: false` — when pagination is
  risk-controlled (~20 posts cap), it launches a real browser so the user can
  solve CAPTCHA. Keep this on.
- `database: true` + `database_path: dy_downloader.db` — dedup/history.
- `increase.post: true` — incremental (only new items) on reruns.
- `folderstyle: true`, `filename_template`, `author_dir` — output layout.
- `transcript.enabled` — needs `OPENAI_API_KEY`; off by default.
- `comments.enabled` — off by default; adds `*_comments.json`.

## Safety / anti-ban posture

This engine is the low-ban-risk choice: your own cookies (real-user traffic),
default 2 req/s rate limiting, exponential backoff retries, and real-browser
fallback for risk control. Do **not** raise `thread` aggressively or disable
`browser_fallback`. For batch archiving, prefer `increase: true` + scheduled
small runs over one giant crawl.

## Troubleshooting

- **Only ~20 posts then stop** → pagination risk control. Confirm
  `browser_fallback.enabled: true`, `headless: false`, and complete the
  CAPTCHA in the popup browser.
- **Cookie expired** → re-run login flow (A or B above).
- **No transcript files** → `transcript.enabled: true` AND `OPENAI_API_KEY`
  set AND `response_formats` includes `txt`/`json`.
- **Re-download something** → delete both the SQLite row AND the local file
  (dual check). Deleting only the DB row won't trigger re-download; deleting
  only the file WILL.
